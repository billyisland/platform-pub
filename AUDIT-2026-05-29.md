# Code Audit — platform-pub

**Date:** 2026-05-29
**Scope:** Full pass, all services. Commit `2ca03ea` (master).
**Method:** Read of the security-critical surfaces (auth, session, crypto, payments, key custody, paywall enforcement, ingest/render) plus an injection sweep across `gateway`, `payment-service`, `key-service`, `shared`.

## Verdict

Strong shipped state. Parameterised queries throughout (no injection surface found), sound crypto primitives, a careful three-phase money flow, idempotent Stripe webhooks, global identity-header stripping at the gateway, sanitised batch ingest, and CI that runs `npm audit`. Three prior audit passes show.

This report focuses on what those passes plausibly missed. One finding to fix before taking real money (live-feed XSS); a cluster of medium auth/reliability issues; minor hygiene. Nothing structural is broken.

## Security

### ~~Medium — live-fetch XSS (highest priority)~~ RESOLVED

The batch ingest adapters all sanitise: `feed-ingest/src/adapters/{rss,activitypub,atproto,email}.ts` route HTML through `sanitizeContent` (verified, with tests stripping `<script>`/`<iframe>`/`<style>`).

The gateway's **live** fetch path does not. `external-items.ts:1151` sets `contentHtml: status.content` and `:1338` sets `contentHtml: s.content` directly from a remote Mastodon/ActivityPub API response. That field is then rendered via `dangerouslySetInnerHTML` in `ExternalCard.tsx`, `VesselCard.tsx`, and `ParentContextTile.tsx`.

A remote instance's own sanitisation cannot be trusted — anyone can run an AP-compatible server returning `<img onerror=...>`. This is a stored/reflected XSS vector.

**Fix:** apply `sanitizeContent` to every live-fetched HTML field, ideally at the single boundary where external HTML enters, so no future caller can forget.

**Resolution:** `sanitizeContent` moved from `feed-ingest/src/lib/sanitize.ts` to `shared/src/lib/sanitize.ts`. Both Mastodon live-fetch paths (parent fetch and thread fetch) in `gateway/src/routes/external-items.ts` now sanitise via the shared module. All adapters updated to import from shared.

### ~~Medium — Google OAuth login-CSRF~~ RESOLVED

`google-auth.ts` (`generateSignedState` / `verifySignedState`, ~line 200): `state` is HMAC-signed but stateless — not bound to the browser, not single-use within its 10-minute window. An attacker can obtain a validly-signed state and fixate a victim into the attacker's Google account.

The cookie-avoidance rationale (Next.js proxy not reliably forwarding `Set-Cookie`) is genuine, but the proper fix is PKCE, or a state nonce stored server-side and consumed once.

Separately: `email_verified` is never checked before trusting `payload.email`. Add it.

**Resolution:** State nonces are now consumed on use (in-memory map with TTL cleanup) — single-use within the 10-minute window. `email_verified` check added; tokens with unverified emails are rejected. `OAUTH_STATE_SECRET` env var separates the state HMAC key from `SESSION_SECRET` (falls back for backward compat).

### ~~Low–Medium — id_token signature not verified~~ RESOLVED

`decodeIdToken` checks `iss`/`aud`/`exp` but not the cryptographic signature. Defensible under OIDC because the token is fetched directly from Google's token endpoint over TLS with the client secret — but fragile: if an `id_token` is ever accepted from the client instead of the direct exchange, it becomes an instant auth bypass (forge any `email`).

**Fix:** verify against Google's JWKS (`google-auth-library` or `jose` + Google keys) for defence-in-depth.

**Resolution:** `decodeIdToken` replaced with `verifyIdToken` using `jose` `jwtVerify` + `createRemoteJWKSet` against Google's JWKS endpoint. Signature, issuer, audience, and expiry are all verified cryptographically.

### ~~Low — dead unsafe renderer~~ RESOLVED

`web/src/lib/markdown.ts:85` `renderMarkdownSync` is an unsanitised regex renderer with attribute-injection XSS (e.g. `[t](https://x" onmouseover=...)` passes the `^https?` check). Currently unused but **exported**, so it's a foot-gun a future caller will reach for. Delete it; keep only the `rehype-sanitize` path.

**Resolution:** `renderMarkdownSync` deleted. Only the `rehype-sanitize` pipeline remains.

### ~~Low — post-sanitise iframe injection~~ RESOLVED

`enhanceEmbedUrls` injects raw `<iframe>` _after_ sanitisation. Safe today because the YouTube ID regex is constrained, but the pattern breaks the moment a less-constrained embed type is added. Move embed handling inside the sanitised pipeline.

**Resolution:** `enhanceEmbedUrls` made private (unexported). A post-enhancement iframe allowlist strips any `<iframe>` whose `src` doesn't match `TRUSTED_IFRAME_PREFIXES` (currently YouTube embed only).

### ~~Low — secret reuse and static internal token~~ PARTIALLY RESOLVED

- `SESSION_SECRET` serves three roles: JWT HMAC, `@fastify/cookie` secret, and OAuth-state HMAC. Separate them.
- Internal services trust one static shared secret with no rotation. Header naming is inconsistent: `x-internal-token` in payment routes vs `X-Internal-Secret` in the scheduler's vault call — confirm both ends agree.

**Resolution:** `COOKIE_SECRET` and `OAUTH_STATE_SECRET` env vars introduced (both fall back to `SESSION_SECRET`). The three cryptographic roles are now separable. Internal header naming inconsistency confirmed as two separate secrets for separate services (working correctly); left as-is since unifying requires deployment coordination.

## Correctness / reliability

### ~~Medium — scheduler crash orphans drafts~~ RESOLVED

`gateway/src/workers/scheduler.ts`: the claim transaction nulls `scheduled_at`, then publish + `DELETE` run _outside_ any transaction. A caught error re-arms the draft (`scheduled_at = now()`), but a hard crash/OOM/redeploy between claim and `DELETE` leaves it with `scheduled_at = NULL` — neither published nor retried, silently lost.

**Fix:** add a `publishing`/`published` status column (or a sweep that re-arms NULL-scheduled, non-published drafts) so a crash is recoverable.

**Resolution:** Claim transaction now sets `scheduled_at = now() + interval '5 minutes'` instead of NULL. On hard crash the draft naturally becomes due again — no orphan possible. Normal error recovery (catch block) still re-arms to `now()` for immediate retry.

### Low — webhook concurrency

`payment-service/src/routes/webhook.ts`: the idempotency guard protects sequential redelivery but not _concurrent_ delivery — the claim (INSERT/SELECT `processed_at`) and the handler are not atomic, so two simultaneous deliveries can both see `processed_at IS NULL`. The settlement/payout handlers have their own conditional-UPDATE claims, so this is largely defence-in-depth, but a `SELECT … FOR UPDATE` on the event row (or a per-`event_id` advisory lock) closes it cleanly.

### ~~Low — dead code in a money path~~ RESOLVED

`payment-service/src/services/settlement.ts` `reserveSettlement`: the `amountPence` ternary is computed and never used; `actualAmount` is the real value, and reduces to `Math.min(lockedBalance, expectedAmountPence)`. Harmless but confusing in a money path — simplify for auditability.

**Resolution:** Dead `amountPence` ternary removed. Simplified to `Math.min(lockedBalance, expectedAmountPence)`.

### ~~Low — stale comments / docs~~ RESOLVED

- Session "refresh at half-life" is wrong: refreshes at 7 days of a 30-day token (~23%), not 15 days.
- README says "7-day lifetime"; code is 30 days. Align.

**Resolution:** Comments in `session.ts` corrected from "half-life" to "refresh threshold (7 days)". `feature-debt.md` description updated.

## Confirmed clean

- **SQL injection:** zero interpolated-value queries across all backend services; `${table}` in key-custody is a controlled enum.
- **Crypto:** AES-256-GCM + XChaCha20-Poly1305 with correct IV/nonce handling; KMS envelope encryption; NIP-44 wrapping correct.
- **Paywall model:** ciphertext lives in the public Nostr event; the content key is only issued server-side after the access check or payment. Exposure rests entirely on key issuance, which is gated.
- **Identity headers:** `stripIdentityHeaders` is a global `onRequest` hook — client-supplied `x-reader-id`/`x-writer-id` cannot reach downstream services.
- **Rate limiting:** present on sensitive routes (auth 5/min, gate-pass 20/min, search 30/min, messages 10/min); global blanket off with documented reason.
- **Settlement `settled_at`:** defaults to `now()` at insert, so reads arriving mid-settlement correctly roll to the next cycle (not a bug).
