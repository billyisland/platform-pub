## What This Is

A publishing and social platform for writers and readers, built on the Nostr protocol. Writers own their identity, audience, and content via custodial Nostr keypairs. Readers pay via a shared "reading tab" (Stripe-based accrual ŌåÆ payout flow).

## Development & deployment

This is a local dev directory on the developer's laptop. Test features at `localhost:3010` (web) and `localhost:3000` (API). To deploy to production: push to git, then the developer SSHs to a remote server and rebuilds there. Claude Code has no direct access to the production server.

### Database migrations

Migrations are numbered SQL files in `migrations/`, applied in numeric-prefix order by the sole runner `shared/src/db/migrate.ts` (`npx tsx shared/src/db/migrate.ts` from the repo root ŌĆö it resolves `migrations/` from cwd; there are no per-service runners). The runner records a sha256 per applied migration and verifies them on every run ŌĆö editing an already-applied file is fatal (corrections go in a NEW migration); the `_migrations.checksum` column is runner-owned bootstrap DDL, never a migration file.

**`schema.sql` is the genesis base, not a derivative.** A fresh DB (dev `initdb.d` and prod) boots from `schema.sql`, *then* `migrate.ts` applies anything newer. So `schema.sql` must already contain every migration's effect **and** seed `_migrations` with every migration filename (so `migrate.ts` is a clean no-op on fresh boot). There is no genesis migration ŌĆö migration `001` ALTERs tables only `schema.sql` creates ŌĆö so the chain can't be replayed from empty; `schema.sql` is load-bearing.

After adding a migration or changing the schema, regenerate `schema.sql` with `pg_dump` from a fully-migrated DB (never hand-edit ŌĆö that breaks canonical round-trip), re-append the `_migrations` seed in the same step, then run the drift guard:

```bash
scripts/check-schema-drift.sh   # 0: seed lists all migrations ┬Ę 3: every migration-created object is in schema.sql ┬Ę 1: migrate is a no-op on a schema.sql DB ┬Ę 2: schema.sql round-trips clean
```

It builds throwaway DBs in the dev Postgres container (read-only w.r.t. your real dev DB) for Checks 1/2; Checks 0/3 are pure-text and need no DB. **CI-enforced** (`.github/workflows/ci.yml` `schema` job). It catches a missing seed entry (Check 0), non-canonical hand-edits (Check 2), and ŌĆö Check 3 (object presence, exit 3) ŌĆö a seeded filename whose **object body** (table/index/type/function/trigger/view) was omitted from `schema.sql`, which Checks 0/1/2 all pass green on (migrate skips the seeded migration, so the object is never created and nothing notices). Check 3 is name-grep against the migration's net-surviving CREATE set (drops + `RENAME TO` netted out in chronological order); its residual gap is **column-level** drift (`ADD COLUMN` missing from `schema.sql`) ŌĆö there the mechanical pg_dump-and-re-append-in-one-step discipline is still what closes it. Never hand-edit the seed line.

## Architecture

Browser ŌåÆ Nginx (80/443) ŌåÆ `/api/*` to gateway, `/` to web. The Next.js app rewrites `/api/*` to `GATEWAY_URL`; the frontend never calls backend services directly.

Orientation only (the cited ADRs / code are the source of truth):

- **Auth**: magic links + Google OAuth (no passwords); httpOnly JWT cookies set by the gateway. `gateway/src/middleware/auth.ts` exports `requireAuth` / `optionalAuth`. Custodial Nostr keypairs: key-custody holds private keys; key-service wraps/issues NIP-44 keys to readers for gated content.
- **Nostr**: articles are kind 30023 (NIP-23) replaceable events signed via key-custody; the platform runs its own strfry relay. Web reads via NDK (`web/src/lib/ndk.ts`). Soft-delete publishes a kind 5 tombstone.
- **Payments**: readers accrue a Stripe tab as they read gated articles; payouts via Stripe Connect (`payment-service/src/services/`). Article access logic in `gateway/src/services/article-access/`; `/articles/:id/gate-pass` is a thin wrapper over `performGatePass()`.
- **Media**: uploaded via `gateway/src/routes/media.ts`, stored in a Docker volume, served at `/media/`; Blossom configured but local is primary. oEmbed proxied in the same route.
- **Editor**: TipTap (`web/src/components/editor/`) with a paywall gate node; markdown via `tiptap-markdown`.
- **Feed & search**: full-text search via `pg_trgm` (`gateway/src/routes/search.ts`); feed ranking `planning-archive/FEED-ALGORITHM.md` (Phase 1).
- **External feeds (Universal Feed)**: RSS/Atom/JSON, external Nostr, Bluesky, Mastodon/Lemmy/threadiverse, and email are ingested by `feed-ingest/` (Graphile Worker + Jetstream listener; adapters in `feed-ingest/src/adapters/`). Data model `external_sources`/`external_subscriptions`/`external_items`, denormalised into the unified `feed_items` timeline by transactional dual-write (each row carries `post_id`, `version`, `biddability_tier`). **External subscriptions are feed-derived** (see Invariants): an `external_subscriptions` row exists iff the source sits in Ōēź1 of the owner's feeds, maintained by `addSource`/`removeSource` in `gateway/src/routes/feeds/sources.ts`; it doubles as the external follow-graph (kind-3 follow list, author Following state). The standalone subscription CRUD + Subscriptions page were retired (migration 116 dropped per-sub `is_muted`/`daily_cap`); `gateway/src/routes/external-feeds.ts` now holds only admin diagnostics, and per-feed mute is `feed_sources.muted_at`. Source authoring via the workspace feed endpoints (`POST /workspace/feeds/:id/sources`, protocols `rss`/`nostr_external`/`atproto`/`activitypub`). Live workspace reads via `GET /workspace/feeds/:id/items` (+ `GET /thread/:postId` projector). Spec: `docs/adr/UNIVERSAL-FEED-ADR.md`; workspace full view: `docs/adr/WORKSPACE-FULL-VIEW-SPEC.md`.
- **Trust graph**: Layer 1 (`trust_layer1`, daily cron) + `trust_polls` compose the `TrustPip` four-state glyph on all feed cards (`feed-ingest/src/lib/trust-pip.ts`). ### Invariants (these are rules, not orientation)

- **Relay outbox**: every write path that publishes a signed Nostr event enqueues into `relay_outbox` inside the caller's transaction (`shared/src/lib/relay-outbox.ts::enqueueRelayPublish`); the `feed-ingest` `relay_publish` worker owns publish + retry. - **Stripe money-moving creates ŌĆö terminal vs ambiguous**: every Stripe call that creates a money-moving resource (`paymentIntents.create` for settlement, `transfers.create` for writer/tribute/publication payouts) uses the three-phase reserveŌåÆcreateŌåÆconfirm pattern with a **row-stable idempotency key**, and MUST wrap the create in try/catch splitting **terminal** (a deterministic rejection that created nothing) from **ambiguous/transient** (connection/API/rate-limit/timeout ŌĆö the resource may have been created before the response was lost). Terminal ŌåÆ mark the row `failed` and release its claims (settlement: drops `reserveSettlement`'s pending-guard + flags `accounts.card_action_required_at`; payout: rolls back claimed reads/accruals via the shared `rollback*PayoutRows`), then return. Ambiguous ŌåÆ **re-throw** so the resume sweep retries with the SAME key (which dedupes a resource that did go through) ŌĆö **never roll back on ambiguous**, or you double-charge (settlement) / double-pay (payout). ## Design system rules

These rules apply to all frontend code. Follow them when writing or modifying components.

### Three-voice typeface system

| Voice                     | Typeface      | Use for                                                                    |
| ------------------------- | ------------- | -------------------------------------------------------------------------- |
| **Literary** (serif)      | Literata      | Article prose, publication names, content previews, the reading experience |
| **Platform** (sans)       | Jost          | UI copy, page titles, body text, descriptions, buttons, display names      |
| **Infrastructure** (mono) | IBM Plex Mono | Labels, metadata, system status, data values, tab pills                    |

Never use serif for platform UI (page titles, settings, admin headings, display names in lists). Never use sans for infrastructure labels. When in doubt, prefer sans over serif for non-literary content.

### The ŌłĆ mark ŌĆö one canonical construction

Every rendered ŌłĆ derives from one geometry (~20.5┬░ leg splay, mitred interior apex, straight crossbar in the upper third at ~84% leg weight), across its three instances: the standalone `ForAllMark` (`web/src/components/icons/ForAllMark.tsx` ŌĆö defines the canonical stance; `size` = rendered **height**, width follows the 0.7 aspect), the `ForallMenu` trigger's idle-ŌłĆ group, and `favicon.svg` (whose rasters `icon-32.png`/`apple-touch-icon.png` are regenerated from the SVG, never edited directly). Don't introduce a new ŌłĆ construction or tweak one instance alone ŌĆö geometry changes re-derive all three from the spec. Typed ŌłĆ characters (ForallCeremony's Literata glyph, "ŌłĆ Traffology") are a different register, exempt. Spec: `docs/adr/LOGO-REFINEMENT-SPEC.md`.

### Use design tokens, not inline sizes

| Token          | Size                                  | Use for                                          |
| -------------- | ------------------------------------- | ------------------------------------------------ |
| `text-ui-xs`   | 13px sans                             | Small UI text, descriptions, secondary copy      |
| `text-ui-sm`   | 14px sans                             | Standard UI text, form values, list items        |
| `text-mono-xs` | 11px mono                             | Small mono data (dates, amounts, tabular values) |

Never hand-roll `font-mono text-[12px] uppercase tracking-[0.06em]` ŌĆö that is `.label-ui`. Never use `text-[13px] font-sans` ŌĆö that is `text-ui-xs`. Never use `text-[14px] font-sans` ŌĆö that is `text-ui-sm`.

### Colour values ŌĆö canonical registry, never a raw hex

Every colour the frontend renders lives as a CSS-variable pair on `:root` ŌĆö `--ah-<slug>` (colour) + `--ah-<slug>-rgb` (RGB triple) ŌĆö defined in `web/src/app/globals.css` from the ordered canonical registry `web/src/lib/palette/registry.ts` (keep both in sync, same order; the registry header documents the contract). Never introduce a raw hex/`rgb()` literal in web code: inline styles reference `var(--ah-<slug>)`, Tailwind colour tokens already resolve to the triples (alpha modifiers like `bg-white/40` keep working), and translucent washes are `rgb(var(--ah-<slug>-rgb) / 0.x)`. A genuinely new colour gets a registry entry + `:root` pair first. Contexts that cannot resolve `var()`: SVG presentation attributes (set `style={{ fill/stroke }}` instead) and canvas (read the triple off `getComputedStyle(document.documentElement)` at draw time ŌĆö see `ProvenanceBar`). Exempt, deliberately hard-coded: Stripe iframe appearance (`CardSetup.tsx`), brand-locked SVGs (Google logo), print CSS. The Palette devtool ŌĆö `web/src/components/devtools/PalettePanel.tsx` + `web/src/stores/paletteDevtool.ts` ŌĆö live-tunes the registry vars (overrides persist in `localStorage` `ah:palette-overrides`). **Operator-only, not user-facing** (GLASSHOUSE-AND-PALETTE-ADR ┬¦III.5): the ForallMenu "Palette" row and the Settings preset-theme picker (`ThemeSection`, now parked) were retired to reclaim a single canonical identity, and existing user overrides are purged once on upgrade (headless, sentinel `ah:palette-purged-v1`). The panel is reached via `?palette` or the `Ctrl+Alt+P` chord. **Do not delete** ŌĆö it remains the operator tuning surface behind the per-feed colour schemes (feature-debt.md ┬¦3), and boot-time hydration of any persisted overrides now lives in the headless `PaletteHydrator` (`web/src/components/devtools/PaletteHydrator.tsx`, mounted at the app root) ŌĆö *that* component is the load-bearing mechanism; never remove it. Per-feed colour **schemes** are a separate axis (`feeds.appearance` / the layout store, `components/workspace/tokens.ts` `PALETTES`), not the override store, and are untouched by the purge. The Glasshouse pane, ŌłĆ chrome (`ink-925`) and trust-pip slugs stay locked out of any theming (`THEME_LOCKED_SLUGS`). The registry + var indirection itself is permanent.

### Workspace colour schemes ŌĆö colour from the palette, never hard-code

Per-feed colour **schemes** in `web/src/components/workspace/tokens.ts`. **A scheme is a COLOURWAY (seasonal character ŌĆö hue / energy / warmth), ORTHOGONAL to light/dark**: the colourway is per-feed, but whether it renders light or dark follows the **global** light/dark toggle (`useColorScheme`). Five colourways: `basic` plus four curated surface families (`spring`/`summer`/`autumn`/`winter` ŌĆö one shared grammar of saturated frame ŌåÆ tinted ground ŌåÆ clean reading surface, DESIGN-TUNING-FINDINGS ┬¦3 superseding the Brazilian-modernist `anil`/`vela`/`caju`). **Dark variants must cohere as a SET, not each on its own** ŌĆö a workspace shows several feeds at once, so the dark `interior`+`card` are a near-shared dark neutral (only faintly hue-tinted) and identity rides the `walls`/bar spine: four clean chromatic hues at matched luminance (~0.38), the Claude-Code "bright accents on a shared dark ground" lesson (DESIGN-TUNING-FINDINGS ┬¦3 addendum, re-tuned 2026-06-22, superseding the earlier contrasting-accent-ground grammar that fought across side-by-side feeds). **Each colourway carries BOTH a light and a dark surface set** (registry slugs; the opposite-of-original variant has a `-dk`/`-lt` suffix ŌĆö Spring/Summer/Autumn authored light-first, Winter dark-first), and `basic` is the two hand-tuned reference palettes `BASIC_LIGHT`/`BASIC_DARK`. They carry **no user-facing display name** ŌĆö the `SchemeSwatch` miniature (which previews the variant for the *current* global mode) is the sole identifier, so the FeedComposer `Colour` control shows no text indicator. **`paletteFor(scheme, dark)`** takes the resolved global-dark flag and returns a `VesselPalette` (`cardBg`/`cardTitle`/`cardMeta`/`interior`/`barText`/ŌĆ” + `isDark`) for that colourway's light or dark variant ŌĆö every field a `var(--ah-ŌĆ”)` reference into the canonical registry; the eight seasonal variants are precomputed (`SEASONAL_PALETTES`). **Quoted-post embeds ride the `walls` surface, not `interior`** ŌĆö `quoteBg`/`quoteText`/`quoteMeta` are derived against walls luminance (Addendum B), so a quote reads as a recessed frame, not a second content well. A curated colourway-variant defines **surfaces only** (walls/interior/card registry slugs); every text colour is **derived by luminance** (`deriveVesselPalette`) between the two reference text ramps, with the semantic flip riding the same switch (crimson ŌåÆ crimson-soft on dark cards) ŌĆö never hand-set a text colour in a scheme, and curate **each variant's** card surface clearly light or clearly dark (mid-luminance defeats both ramps). The scheme is **feed character**: persisted server-side (`feeds.appearance.scheme`, migration 112; cycled through the FeedComposer `Colour` AppearanceControl ŌĆö GLASSHOUSE-AND-PALETTE-ADR ┬¦III.4), mirrored into the layout store's legacy `brightness` field as the per-device cache/fallback. The colourway id list must mirror `FEED_SCHEME_IDS` in `gateway/src/routes/feeds/crud.ts`; retired ids (incl. the former mode-fixed `primary`/`dark` ŌåÆ `basic`) are migrated on read by `normalizeBrightness`'s alias map (`SCHEME_ALIASES`), so renaming schemes needs no DB backfill. **Density rides the same model** (`feeds.appearance.density`, no DDL; `FEED_DENSITIES` in the gateway must mirror the `Density` type in `tokens.ts`), as do `feeds.hidden` + `feeds.sort_rank` (migration 113): hide and rank are feed character, never per-device layout state, and the numeral is derived 1..N over **visible** feeds in `sort_rank` order ŌĆö one rule on every surface (MOBILE-LAYOUT-ADR ┬¦V/┬¦VII).

- **Any component inside a themed vessel interior or card takes the palette** (`palette: VesselPalette` prop, or `paletteFor(brightness)`) and colours every text/background ŌĆö **including hover and active states** ŌĆö from it. ### Global light/dark mode (per-device theme)

A sitewide **Light / Dark / System** appearance toggle, per-device (localStorage `ah:color-mode`, default `light`), modelled on the type-size control: store `web/src/stores/colorScheme.ts`, headless `ColorSchemeHydrator` (mounted in `LayoutShell`), settings UI `account/ColorModeControl`, plus a blocking inline script in `app/layout.tsx` (no white flash). It toggles `html.dark`.

**Mechanism ŌĆö invert-at-root + feed "light islands":**
- Dark mode is an `html.dark { ŌĆ” }` block in `globals.css` that **inverts the canonical NEUTRAL ramp** at the document root (re-declares the `-rgb` triples). Because all global chrome reads those vars, the whole shell flips automatically: workspace ground, reader/profile/source/tag surfaces, **every Glasshouse overlay** (settings/dashboard/messages/admin), dropdowns/popovers. Only neutral chrome slugs flip (`DARK_SLUGS` in `web/src/lib/palette/island.ts`: `ink`, the greys, `white`, `glasshouse`, `glasshouse-well`, `bone`/`bone-bright`, `off-white`, `cream*`, `nav-grey`). Accent (`crimson*`), semantic (`trust-*`, `danger`/`vouch`, `klein-blue`), already-dark (`true-black`, `ink-925/900/850`, `stone-*`, `wall-grey`) and every per-feed season slug are **never inverted**.
- **Desktop feed vessels render their colourway in the global mode's light/dark variant** (the scheme is character, light/dark is global ŌĆö see *Workspace colour schemes*): each `Vessel` reads `useColorScheme().dark` and calls `paletteFor(brightness, dark)`. The vessel **still** carries `LIGHT_ISLAND_STYLE` ŌĆö but now the island's job is only to keep the *neutral text slugs* the derivation references (bone/ink/white/stone/ŌĆ”) resolving canonical regardless of mode, so the derived ramps stay deterministic; the **variant** (not the island) supplies the light/dark. Season surface slugs (`spring-*`/`winter-*`/ŌĆ”) are NOT in `DARK_SLUGS`, so they never invert ŌĆö the variant choice is the only thing that flips them. (Profile/source/tag content carries no colourway and uses `globalContentPalette(dark)` OUTSIDE any island, relying on the html.dark inversion; **mobile feeds DO carry the colourway** and are islanded like desktop vessels ŌĆö see below.) The **ForallMenu disc + dropdown** also carries the island (locked nav chrome: the dropdown stays light; the wordmark, a sibling over the floor, flips via `var(--ah-ink)`) and the FeedComposer **scheme swatch** (which previews the colourway's variant for the *current* global mode via `paletteFor(scheme, dark)`; the island only keeps `basic`'s neutral slugs clean in the preview). **The disc itself deliberately inverts under dark mode** ŌĆö it stays islanded (tokens resolve to canonical light) but explicitly swaps to a light `bone` fill + dark `ink-925` glyph (`discBg`/`discGlyph` in `ForallMenu.tsx`, gated on `useColorScheme().dark`), the photo-negative of the light-mode dark disc; the dropdown is unaffected. Keep `DARK_SLUGS` in sync with the `html.dark` block. **`LIGHT_ISLAND_STYLE` must redeclare BOTH `--ah-<slug>` and `--ah-<slug>-rgb` for every slug** ŌĆö the colour var is declared at `:root` as `rgb(var(--ah-<slug>-rgb))` and resolved *there* (where `html.dark` set the dark triple), so re-pinning only the `-rgb` triple on a descendant does **not** change the inherited resolved `--ah-<slug>` (the dark-on-dark feed + dark-disc-glyph bug, fixed 2026-06-21). Pin the colour var for `var(--ah-<slug>)` consumers and the triple for `rgb(var(--ah-<slug>-rgb) / ╬▒)` alpha consumers.
- **Mobile feeds render the per-feed colourway in the global mode's variant**, same as desktop: `WorkspaceView.renderFeedContents` uses `paletteFor(brightness, dark)` for both surfaces, and the mobile per-feed pages carry `LIGHT_ISLAND_STYLE` (`MobileWorkspace.tsx`) so the derived text resolves canonical. The mobile **bar** above the pages is NOT islanded ŌĆö it is global chrome and inverts with the toggle.
- **Content that should follow the global theme** (profile/source/tag content-log cards) uses `globalContentPalette(dark)` from `tokens.ts` (the `BASIC_LIGHT` palette, whose slug refs invert, with `isDark`/crimson **plus the mode-specific stone tones** corrected for dark ŌĆö `cardStandfirst`ŌåÆ`stone-300`, `nameLabel`ŌåÆ`stone-350`, because `stone-*` is **not** in `DARK_SLUGS` so it never inverts, and the light-mode `stone-600` would otherwise render dark-on-dark) ŌĆö *outside* any island, so the inversion takes effect. Article prose (`prose`) gets a `html.dark .prose` `--tw-prose-*` override so the body/lists/rules invert too.

When adding a surface, decide: is it **global chrome** (let it invert ŌĆö do nothing) or a **feed vessel** (island it)? Never island a profile/reader/overlay surface (it would stay light in dark mode). Spec: `docs/adr/GLASSHOUSE-AND-PALETTE-ADR.md` (global theme section).

### No hairlines, no outlines, no single-pixel anything

Absolute, sitewide invariant. **The site never renders a 1px line, anywhere, ever** ŌĆö no hairline dividers, 1px borders/outlines/rings, 1px line elements, `<hr>`, or `box-shadow` used as a line. No exception for "dense UI chrome" (dropdowns, menus, popovers, settings rows) ŌĆö that loophole is closed.

Banned, never to be introduced:

- Tailwind 1px-resolving utilities: bare `border` / `border-t|b|l|r|x|y`, `border-[1px]`, `divide-x` / `divide-y`, `ring-1` / `ring-px`, `outline-1` / `outline-px`, `h-px` / `w-px` / `h-[1px]` / `w-[1px]`.
- The raw `<hr>` element. Any `hairline` color token used to draw a line.

Instead: **separation** ŌåÆ whitespace (`space-y-*`, `gap-*`) and feed/thread rhythm; **major divider** ŌåÆ `.slab-rule-4` (never hand-rolled `h-[4px] bg-black`); **enclosure/emphasis** where a real border is unavoidable ŌåÆ `>= 2px` (the 2px a11y focus outlines in `globals.css` are correct, not hairlines).

### One post per card

A card renders exactly **one** post ŌĆö never two or more fused. Absolute, sitewide. Relationships between posts (parent ŌåÆ reply, a burst of replies) are expressed by **expanding a card into the thread** (ancestors above, focal, replies below), never by inlining another post's body into a card. Banned fusions, not to be reintroduced:

- **Parent-context tile** ŌĆö a collapsed reply must not inline its parent above the byline. (Reply context comes from thread expansion.)
- **Reply groups** ŌĆö a burst of replies sharing a parent must not collapse into one multi-reply card. The gateway feed query emits each reply as its own item; there is no `reply_group` envelope.

Quote embeds are a distinct grammar (a quote *is* one post referencing another) and are exempt. The Quote action is on **all** card tiers, native and external: quoting an external post publishes a native note embedding the origin as a `QuotedEmbed` mini (author ┬Ę source ┬Ę excerpt, linked to permalink) ŌĆö no NIP-18 `q` tag; the reference rides `notes.quoted_post_id`/`quoted_url`/`quoted_source` (migration 102). See UNIVERSAL-POST-ADR.

### Feed card chassis

All three feed card types (`ArticleCard`, `NoteCard`, `ExternalCard`) share a unified grammar:

- **Left bar**: 4px solid, full height. Black (`#111111`) native, crimson (`#B5242A`) paid, grey-300 (`#BBBBBB`) external. `borderLeft` inline + `paddingLeft: '24px'`.
- **Byline row**: mono-caps 11px (`font-mono text-[11px] uppercase tracking-[0.06em] text-grey-600`). Order: TrustPip ┬Ę Author ┬Ę middle-dot ┬Ę timestamp ┬Ę (optional price or protocol badge). While trust is parked the `TrustPip` is a **decorative dot** (`aria-hidden`, no title/cursor) that opens nothing ŌĆö the click-to-open `PipPanel`/`PipTrigger` are **dormant, not deleted** (restore by reinstating the `PipTrigger` branch + `onPipOpen` in `PostByline`). The per-feed author/source actions it used to host moved to the byline hover panel (next bullet).
- **Byline routing** (same rule on cards and on every expanded parent/reply): a byline links to the internal all.haus surface, never the origin platform. Native ŌåÆ writer profile (`/{username}`). **Tier-A/B external** (carrying an `external_authors` record ŌĆö nostr pubkey, atproto DID, activitypub actor) ŌåÆ `/author/:authorId` (UNIVERSAL-POST-ADR ┬¦4.4). **Tier-C/D external** (rss/email, `post.author.id` null) ŌåÆ **plain text**, no link. The only route out to origin on a card is the source-attribution line. Every linked byline (native + A/B) opens a debounced, session-cached `AuthorModal` (`type="author"` ŌåÆ `GET /author/:id/profile`); inside it the display name links internally (`profilePath`) and ŌĆö A/B only ŌĆö the `@handle` links out to origin (`externalUrl`: bsky.app / actor URI / njump.me, built by `gateway/src/lib/author-resolve.ts::buildExternalProfileUrl`). The modal is the one byline affordance routing to origin. The hover panel also hosts the **per-feed VOLUME control** (`SourceVolume`, `web/src/components/feed/SourceVolume.tsx`) ŌĆö the 5-step weight + RANDOM/TOP sampling, shown only for **sources you follow**: native authors via the `/author-volume/:pubkey` route (pubkey threaded from the byline), external sources via `patchSource` on the resolved `feed_sources` row (the same universal by-feed-source-id path the FeedComposer uses; absent row ŌćÆ not in this feed ŌćÆ no control). This replaced the parked pip panel's volume bar. Reachability is hover-only (no touch path ŌĆö mobile tunes volume in the FeedComposer).
- **Action row**: mono-caps 11px (`tracking-[0.02em]`). `Reply` opens the compose overlay; `Quote`, `VoteControls`, `BookmarkButton`, `ShareButton` per type.
- **No avatars** in card bodies ŌĆö left bar + pip + mono-caps name carry identity.
- **Feed rhythm**: each `PostCard` carries its own `marginBottom` (no wrapping `space-y`), per `GAP_PX` in `web/src/lib/post/level-spec.ts` ŌĆö **8px** between independent items, **5px** within an expanded conversational chain. (The legacy `components/feed/` card stack is retired ŌĆö FEED-RETIREMENT Slice 7; only the shared `AuthorModal` survives there. Legacy `PlayscriptThread` still uses `space-y-[32px]`.)

### Reply threads (playscripts)

Threads render flat and chronological as a transcript ŌĆö never nested indentation, left borders between replies, quote-of-parent blockquotes, or avatars. Component surface: `PlayscriptReply` + `PlayscriptThread` in `web/src/components/replies/`; `ReplySection.tsx` flattens the nested API tree into `PlayscriptEntry[]`.

- **Step-in**: thread container indented 32px once (`ml-8`) from the parent's content column ŌĆö the only indentation in the system.
- **Rhythm**: 32px (`space-y-[32px]`).
- **Speaker line**: mono-caps 11px. `TrustPip` ┬Ę bold Jost name ┬Ę colon. Own replies read `YOU:` with no pip (the asymmetric 16px left-jog is deliberate). Non-adjacent parent: prefix `ŌåÆ PARENT:` in `grey-400` + 16px gap, then `NAME:`.
- **Dialogue line**: Jost 14.5px (`text-[14.5px]`), `leading-[1.55]`, black, at `mt-1`.
- **Vote count**: `VoteControls` pinned top-right, aligned to the first dialogue line.
- **Action row**: `time ┬Ę REPLY ┬Ę DELETE ┬Ę REPORT`, mono-caps 11px `text-grey-400`, revealed on hover/focus (optional `#fafaf7` tint).
- **Pagination**: first 10 entries + `SHOW N MORE REPLIES` (mono-caps, grey-400, underline on hover).
- **Workspace focal node + parents-above**: an expanded conversation reads strictly top-down ŌĆö focal in the middle, **ancestor chain above** (one flat `ml-8` indent, never a deepening cascade), descendants below. No separate pinned card or duplicated byline. **Clicking any ancestor/reply re-roots in place** (repeatable); **clicking an external quote tile re-roots onto the quoted post** (gateway mints a context-only `feed_items` twin on demand) ŌĆö the same click on a **collapsed feed card** expands the quoted post directly as the focal of a fresh conversation, distinct from a body click that expands the host (UNIVERSAL-POST-ADR, 2026-06-14); **clicking the focal collapses the card**; `Ōåæ Full conversation` returns to the opened item. **Rich focal, light context** (native + external): the focal always renders its full rich body (content, media, action row) ŌĆö including after re-rooting; ancestors/replies render as lightweight playscript. **No left bar is ever drawn on a focal node.** Native re-roots from the in-memory tree (`renderFocalNode`); external re-roots fetch full data via the thread endpoint's `focus` field. Byline routing as in the feed chassis. Wiring (`ConversationView`/`useConversation`; `ExternalAncestorRail`/`useExternalThread`) and full behaviour: `docs/adr/CARD-BEHAVIOUR-ADR.md` addendum (2026-05-30).
