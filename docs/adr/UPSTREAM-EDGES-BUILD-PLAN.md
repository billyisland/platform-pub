## Phase 5 — as built (2026-06-24)

A uniform recursive generalisation of the shipped Phase-3 money model — every node (author at depth 0, every inspirer below) now runs the *same* carve. Shipped dark behind `TRIBUTES_ENABLED`; the production money flag stays OFF (compliance residual #1, unchanged — C6 opens no new gate). Built in the planned 5a → 5b → 5c order.

- **5a schema (migration `128_tribute_chains.sql`).** `tributes.parent_tribute_id` (NULL = root; non-NULL = child of that tribute's share) + `tributes.depth int CHECK (0..8)` (the C4 cap, set in the route, CHECK-backstopped). The ceiling trigger `tributes_validate_write` generalised from article-scoped to **parent-scoped** (C3): roots compete among the article's roots, children among siblings under one parent — each node keeps ≥10% of *its own* inflow. C5's swept-return vehicle is the one new money plumbing: `tribute_accruals.author_return_payout_id` (single FK → `writer_payouts`) folded into a polymorphic **`swept_return_payout_id` + `swept_return_kind` ('writer'|'tribute')** pair (soft-ref, since a return now targets *either* a writer payout (root → author) or a tribute payout (child → parent inspirer)), with an all-or-nothing consistency CHECK; the old single-FK column + its partial index are migrated then dropped. Index `(parent_tribute_id)`. `schema.sql` regenerated + seed re-appended; all four drift checks green (307 objects).
- **5a authoring (`gateway/src/routes/tributes.ts`).** `POST /tributes` takes an optional `parentTributeId`; the ownership check generalises from "owns the article" (root) to "**is the live beneficiary of the parent**" (child, C1 — only a `live` tribute may spawn children, validated by `loadLiveParent`). A child sets `author_account_id` ← offerer (= the parent's beneficiary, uniform with roots), `article_id`/`depth` ← derived from the parent row, rejects depth > 8, and serialises on a **parent-scoped** advisory lock. `citationEdgeId` is roots-only. `POST /tributes/:id/consent` returns `articleId`/`percentageBps` so the client can open the upstream child composer (the combined accept-and-offer, C1). `GET /articles/:id/tributes` projects `parentTributeId`/`depth`.
- **5b money (`payment-service/`).** `confirmSettlement` replaces the single-level `INSERT … SELECT` with a `WITH RECURSIVE` walk over the tribute tree ⋈ the just-settled reads: each node freezes its **gross** = `read_net × (∏ bps along its path)`, the path product carried as **NUMERIC** (so it can't overflow bigint at depth 10000⁸) and floored once per (node, read). `runPayoutCycle` / `reserveWriterPayout`: the author (depth 0) now carves **ROOT** accruals only (state-agnostic) and folds **root** swept returns via `swept_return_payout_id` kind `'writer'`. `runTributePayoutCycle` / `reserveTributePayout`: each inspirer node is now a co-earner paid `Σ(its released accruals) − Σ(its direct children's accruals on those reads) + Σ(its direct children's swept returns)`, claiming child swept shares via kind `'tribute'` (C5) — the author's carve+swept-return shape one level down. The eligibility set generalised to the inspirer mirror of the author's `base ∪ ret` (released-unclaimed ∪ children-swept-owed). Display carve: `getWriterEarnings`/`getPerArticleEarnings` subtract **root** held|released|paid accruals (deeper shares live within a root's gross — telescoping); `reservedPence` (compliance #4) recomputed uniformly as Σ(held|released) accruals where `tribute.author_account_id = X` — capturing both an author's reserved-from-earnings (roots) and an inspirer's reserved-pending-onward-redirect (children), one query for the whole tree.
- **Conservation verified.** The gross-freeze + carve telescopes exactly: `author + Σ(every node's retained) == read_net`, independent of rounding (each node's retained is *defined* as gross − Σ direct children gross). Smoke-tested numerically (read_net 1000, root 20% → child 50% → grand 50% ⇒ 200/100/50 gross, retained 800/100/50/50 = 1000). All new SQL (recursive walk, level-aware reserve, candidate set) validated against the live schema.
- **5b reconcile/guards (`scripts/reconcile-ledger.sql`).** The Phase-3 global "Σ paid accruals == Σ tribute_payout" no longer holds (paid accruals sum to *all* gross; the ledger sums root gross), so A10 generalises to **A10a** (per-node `paid_out == node paid gross − direct children paid gross`), **A10b** (tree telescoping `Σ tribute_payout ledger == Σ root paid gross`), and **A10c** (swept_return_kind ↔ depth + payout-table consistency, incl. the orphan branch for the new soft-ref claim). A9/A11 unchanged. **No new `LedgerTriggerType`, no new adjacency MARKER/REGISTRY entry** (the settlement freeze stays one `INSERT INTO tribute_accruals` driven by a recursive SELECT; payouts stay `INSERT INTO tribute_payouts`); adjacency guard re-run green.
- **5c rendering (`web/src/components/article/UpstreamEdges.tsx`).** The Tributes section renders the **tree** (`renderTributeForest` walks parent→children, indenting by depth; a child reads "X% of *that share* …" with a `↳` and a depth-aware return phrase). The inspirer's proposed offer gains the third option **"Accept & pass a share upstream"** (gated on `depth < 8`), which consents then opens a `key`-remounted child `TributeComposer` (parent-context copy, `parentTributeId` carried to `POST /tributes`) — available to any accepting inspirer, not only the article author. The "reserved, pending redirect" indicator already renders for any account with `reservedPence > 0`, so it now covers inspirer nodes too. API client: `parentTributeId`/`depth` on `TributeView`, `parentTributeId` on `CreateTributeInput`, `articleId`/`percentageBps` on the consent response.

## Phase 3 — as built (2026-06-24)

Shipped behind `TRIBUTES_ENABLED` (migration `127_tribute_money.sql` + the payment-service settlement/payout changes + the gateway display carve). No money moves until the operator sets the flag (compliance gate). Concrete decisions and the one deliberate deviation from the plan's prose:

- **Conservation realization (deviation, dictated by the named reconcile invariant).** The plan's prose said the author carve subtracts `held|released|paid` "plus swept accruals returned." That wording double-counts a share that is swept *before* its read is paid (read and accrual reach terminal state in either order). The plan's own named reconcile — **`author-share + Σ accruals == read net`** — says the carve subtracts **Σ(ALL accruals on the read)**, no state filter. So as built: the per-read carve subtracts every accrual on the read; each accrual's disposition is the second leg — `released → paid` (inspirer) or `swept → returned` (author). The author DISPLAY paths still carve `held|released|paid` only (swept/returned are the author's), so money and display converge on `author = read_net − (held|released|paid)`. This is the only ordering-safe form; documented in migration 127's header.
- **Migration 127** adds the `returned` terminal state to `tribute_accruals` (a swept share returned to the author), two exactly-once claim columns (`tribute_payout_id` for released→paid, `author_return_payout_id` for swept→returned, mirroring how `read_events` claim a `writer_payout`), the `tribute_payouts` table (the inspirer-payment record, same crash-safe three-phase shape as `writer_payouts`), and widens `ledger_writer_earnings` to count `tribute_payout` (guard #5). `LedgerTriggerType += 'tribute_payout'`.
- **Settlement apportionment** (`confirmSettlement`, gated on `tributesEnabled()`): one `INSERT … SELECT` after reads advance to `platform_settled` freezes a `tribute_accruals` row per (live|proposed tribute, settled read) = `FLOOR(read_net × bps / 10000)`, dropping 0-pence rows; `ON CONFLICT DO NOTHING` for webhook-retry idempotency. State is `released` when the tribute is already `live` (no held→released flip is coming for post-consent reads), else `held`.
- **Author carve + swept-return** (`runPayoutCycle`): eligibility + `reserveWriterPayout` compute `net = base per-read/upvote net − Σ(ALL accruals on the settled-unpaid reads) + Σ(swept shares owed back)`; swept accruals are claimed under `author_return_payout_id` at reserve, advanced to `returned` at complete, rolled back on a failed transfer. `candidates = base ∪ ret`, so a writer whose only balance is a swept return is still eligible. The author's `writer_payouts.amount_pence` (= the Stripe transfer = the `writer_payout` ledger entry) already nets the carve and folds in the returns — no separate transfer, no mirrored debit (A4 holds).
- **Inspirer payout** (`runTributePayoutCycle`, gated, runs after the writer + publication cycles): per live tribute with released, unclaimed accruals and a Connect-onboarded inspirer — reserve a `pending` `tribute_payouts` row + claim the accruals, transfer (stable key `tribute-payout-${id}`), then flip `initiated`, advance accruals `released → paid`, post **one** `tribute_payout` ledger entry (account = inspirer, counterparty = author). No threshold (the author's read already cleared £20 to settle). `resumePendingTributePayouts` retries crashed-pending rows with the same key.
- **Author display carve-out** (`getWriterEarnings`, `getPerArticleEarnings`, `my-account.ts`): each writer-earning figure subtracts `held|released|paid` accruals on the read via the shared per-read-net helper + a tribute-accrual aggregate, so a tributed author's dashboard no longer over-reports. `publications/revenue.ts` is untouched — D1 forbids a tributed piece inside a publication, so its reads never carry accruals.
- **Shared helper** `shared/src/lib/per-read-net.ts` (`perReadNetPence` + `readNetSql`) is the single net definition the settlement, payout, and display paths all consume (the formula was hand-duplicated across ~12 SQL sites). Unit-tested in `shared/tests/per-read-net.test.ts`.
- **Guards:** adjacency `MARKERS += INSERT INTO tribute_accruals|tribute_payouts`, `payout.ts` floor 2→3 (the new `tribute_payout` recordLedger), `settlement.ts` stays 1 (accruals are off-ledger). `reconcile-ledger.sql` gains A9 (tribute_payout↔tribute_payouts), A10 (`Σ paid accruals == Σ tribute_payout`), A11 (no ledger entry references `tribute_accruals` — held shares stay off-ledger, guard #7), the A6 orphan branch, and the B2 inspirer-payout anchor. All four schema-drift checks green (306 objects); adjacency + reconcile green; payment-service (48) + shared (83) tests green; root promise-safety lint 0 errors.
- **Reserved indicator (compliance condition #4) — shipped 2026-06-24.** `getWriterEarnings` now returns `reservedPence` (Σ held+released accruals on the author's reads — the in-flight carve, excluding `paid`/`swept`/`returned`), proxied through the gateway to the web `WriterEarnings` type. `BalanceHeader` renders a "Reserved for tributes — reserved from your earnings, pending redirect to the sources you credited; returns to you if an offer isn't taken up" line when `reservedPence > 0`, gated on `NEXT_PUBLIC_TRIBUTES_ENABLED` (dark otherwise). Honest "reserved, pending redirect" framing, never "paid to X".
- **Still dark:** the production `TRIBUTES_ENABLED` / `NEXT_PUBLIC_TRIBUTES_ENABLED` flags are OFF. Enabling them remains the compliance owner's call (memo residual #1, the platform-wide Stripe-PI baseline).

## Decisions resolved here (baked into the plan)

These were open in the ADR; settled for v1:

- **D1 — Publication × tribute: disallow.** A tributed piece may **not** be inside a publication, and a publication article may not be tributed. Enforced by validation in `POST /tributes` (reject when `articles.publication_id IS NOT NULL`) and in the publication article-add path (reject when a live/proposed tribute exists). This sidesteps the double-splitter (publications split `read_events` revenue at payout; tribute carves it at settlement — composing them means teaching `reservePublicationPayout` to subtract accruals). Composition is deferred to Phase 4. *(ADR Edge cases › Publication interaction.)*
- **D2 — Dispute stake is a real tab charge.** Modelled as a `−amount` debit on the disputant's `reading_tabs.balance_pence` (same shape as `vote_charge`), collected at their next settlement, refunded as a `+amount` tab credit on withdrawal. "Never forfeited" = always reclaimable by withdrawing; an un-withdrawn stake stays held (charged + parked), never lost. No new escrow primitive. *(ADR Decision 11.)*
  - **Collectibility caveat (known soft edge):** the stake is only ever *collected* when the disputant next settles a tab, which requires a Stripe payment method. A disputant with no payment instrument — who never reads paid content — never settles, so the debit sits on a (now-permitted) negative balance and is never realised. The anti-spam friction is therefore real for paying readers and **soft for drive-by accounts** (exactly the spam population). v1 accepts this; **require a payment method on file to file a *staked* (third-party) dispute** as the cheap hardening if abuse appears. The no-stake cited-author/credited-party path is unaffected.

## Grounding (verified against the codebase 2026-06-22)

The plan depends on these facts; if they change, revisit:

- **Per-read writer net is computed at *payout*, not stored** — `amount_pence − FLOOR(amount_pence * platformFeeBps / 10000)`, floored **per row** (`payout.ts` `runPayoutCycle`/`getWriterEarnings`; `platformFeeBps` default 800, `shared/src/db/client.ts:91`). Settlement computes the platform fee on the **tab aggregate** only (`reserveSettlement`, `settlement.ts:196`), and `confirmSettlement` (`settlement.ts:356`) bulk-advances many `read_events` to `platform_settled`. ⇒ Apportioning "at settlement" is a genuinely new step: compute per-read net at `confirmSettlement` for tributed reads. **Factor the per-read-net formula into one shared helper.** This formula is currently *hand-duplicated across ~12 sites in three files* — `payout.ts` (runPayoutCycle, getWriterEarnings, …), `gateway/src/routes/publications/revenue.ts`, `gateway/src/routes/my-account.ts` — with no central wrapper. The helper must back **both** the money paths (`confirmSettlement` + payout cycle) **and** the read-side display paths (Phase 3 author-display carve-out), or conservation and the author's dashboard drift apart.
- **Un-onboarded earnings hold implicitly** — there is no holding table; `platform_settled` reads with `writer_payout_id IS NULL` simply wait, because the payout eligibility query filters `stripe_connect_kyc_complete = TRUE AND stripe_connect_id IS NOT NULL`. ⇒ A released tribute accrual must be payable **even when the author never onboards**, so the inspirer payout is a **dedicated sweep** keyed on `tribute_accruals.state='released'`, not a rider on the author's read sweep.
- **Resolver** — `POST /resolve` (`gateway/src/routes/resolve.ts`, logic `gateway/src/lib/resolver.ts`) with `context:'invite'` (native-only, `skipExternal`) returns a `native_account` match carrying `account.id` for members, `external_source`/`rss_feed` otherwise. This is the inspirer-identification call.
- **Co-earner analog** — `computePublicationSplits` (`payout.ts:56-111`) + `publication_payout_splits`/`publication_article_shares` (`migrations/038_publications.sql`) is the existing multi-party split, but it runs at **payout** on a publication's aggregated pool and only ever pays resolved member accounts. Tribute's per-read-at-settlement + held-for-a-non-account is different enough that `tribute_accruals` is its own side-table (ADR Decision 3). General per-read co-earner attribution stays **deferred**.
- **Notifications** — no central helper; raw `INSERT INTO notifications (recipient_id, actor_id, type)`. Template: `gateway/src/routes/publications/members.ts:97`. Dedup is via partial unique indexes (`migrations/014`,`019`); a new `type` needs its own index to dedup.
- **Ledger funnel** — every money movement posts through `shared/src/lib/ledger.ts::recordLedger(client, …)` in the caller's txn. Append-only is DB-enforced (migration 119 row guard, 124 TRUNCATE guard). `reading_tabs.balance_pence` may go negative (migration 124) — **never re-introduce a clamp**.
- **Next migration number: 125.**

## Phase 1 — Edges (credit, citation, dispute). No compliance gate.

Money-free except the disputant's own stake (D2), so it ships independently of the tribute compliance question. Establishes the ledger-guard wiring early.

**Schema (migration 125):**
- `credit_edges`, `citation_edges`, `dispute_edges` per the ADR sketch, with these fixes:
  - Drop `target_kind`; adopt the `citation_edges` convention (NULL `target_protocol` = native, non-NULL = that external protocol). Add a CHECK tying `target_protocol`/`resolved_account_id`/`target_external_id` consistency.
  - `dispute_edges`: keep the single-target CHECK `((citation_edge_id IS NULL) <> (credit_edge_id IS NULL))`.
  - Indexes: `(article_id)` on each edge table; `(citation_edge_id)`/`(credit_edge_id)` on disputes; for dispute-privilege lookup, `(source_author_pubkey)` on citations **and** `(resolved_account_id)` on credits (the credit-disclaimer match, `POST /disputes` below).
- Add `dispute_stake`, `dispute_stake_refund` to `LedgerTriggerType`; add both to `ledger_reader_balance` (guard #4) in the same migration.

## Phase 5 — Tribute chains (recursive re-division). SCOPED, NOT YET STARTED.

Specified 2026-06-24 in `UPSTREAM-EDGES-ADR.md` › *Tribute chains (recursive re-division)* (and Decision 13, C1–C6). An offered inspirer's third reply — beside accept/decline — is **accept and pass a share of their own share further upstream**, repeating to a bounded depth. A tribute gains an optional parent; the shipped Phase-3 money model (gross-freeze at settlement, carve-at-payout) applies **recursively at every level** — it subdivides an already-consented share, so it opens **no new compliance question** (the held-funds position of `UPSTREAM-EDGES-TRIBUTE-COMPLIANCE.md` recurses one level up the tree, ADR C6) and most of the system recurses unchanged. This is a **net-new feature on top of shipped Phases 1–4**, build-ordered after them; it does **not** block, and is not blocked by, the still-open Phase-3 money-flag compliance sign-off (the recursion is dark behind the same `TRIBUTES_ENABLED` flag).

**The keystone reframing:** the author was always the depth-0 case of "a party that carves its direct children's shares off its inflow." Phase 3 only ever had depth 0 (author carves its root tributes). Phase 5 lets the tree go deeper and makes every node — author included — run the *same* carve. So the work is a **uniform generalisation of the shipped model, not a new mechanism** (ADR feasibility verdict: feasible, moderate). The two genuinely new things are (a) the parent pointer + parent-scoped authoring, and (b) C5's swept-share-returns-to-the-*parent* (not always to the author) payout vehicle. Everything else is "subtract direct children instead of subtract roots."

### What already recurses for free — do not re-touch

The contact pipeline (in-app notification + email + `/tribute/claim`), the consent/decline lifecycle, the 60/30-day window worker (`tribute-sweep.ts`), comp-read grants, the `tribute_offer_received` notification (now actor = any live parent's beneficiary, not only the author), proposed-offer withdrawal, the `tribute_payout` `LedgerTriggerType` + its ledger trigger, the per-article D1 publication guard, and D1 itself (one non-publication piece carries the whole chain). The offerer generalises from "the article author" to "the article author **or** a live parent inspirer" — that single substitution is most of the authoring change. **No new `LedgerTriggerType`, no new adjacency `MARKERS` entry** (the settlement freeze stays an `INSERT INTO tribute_accruals`, just driven by a recursive SELECT; the parent-return is still a `tribute_payout`/`writer_payout` entry).

### Schema (migration 128)

- **`tributes.parent_tribute_id uuid REFERENCES tributes(id)`** — NULL = a *root* tribute (today's behaviour: carves the piece's writer-side net). Non-NULL = source-of-funds is the **parent tribute's share**, not the piece net. The structure is a tree rooted at the piece; each root→leaf path is one chain. `percentage_bps` keeps one meaning at every level — *the share of the parent's inflow* (the piece net being the implicit parent inflow for roots), so **no column rename**.
- **Generalise `author_account_id`'s meaning (no DDL):** for a root it is the article author; for a child it is the **parent's beneficiary** (the offerer who is redirecting *their* share) — i.e. the parent tribute's `resolved_account_id`. It stays "whoever is paying this share out of their slice" and remains the `tribute_payout` ledger counterparty. Document the widened meaning in the migration header; the column and FK are unchanged.
- **`tributes.depth int NOT NULL DEFAULT 0`** — derived as `parent.depth + 1` on insert (cheaper than a per-insert recursive walk; the authoring route sets it from the parent row it already loads). Enforce the cap (**8**, C4) in the route and as a CHECK/trigger backstop. Bounds the recursive settlement walk and stops sub-penny dust chains.
- **Generalise the ceiling trigger `tributes_validate_write` from article-scoped to parent-scoped (C3):** the Σ-`percentage_bps` ≤ 9000 sum currently groups by `article_id` over `status IN (proposed,live)`. It becomes grouped by the **parent stream**: roots (parent NULL) still group by `article_id` *among roots*; children group by `parent_tribute_id`. Each node thus retains ≥10% of *its own* inflow, so a share is never fully drained past the node that accepted it. The D1-forward branch and the `articles` D1-reverse trigger are unchanged.
- **C5's swept-return vehicle — the one new money column.** Today a swept share folds back into the **author's** `writer_payouts` run via `tribute_accruals.author_return_payout_id → writer_payouts(id)`. A deeper child's swept share must instead fold into its **parent inspirer's** `tribute_payouts` run. A single strict FK can't reference both `writer_payouts` and `tribute_payouts`. **Decided (ADR C5): one generic "return to the parent's payout" path, author = depth-0 case — not a parallel column.** Schema realisation: a single claim column `swept_return_payout_id uuid` **+ a `swept_return_kind text CHECK (swept_return_kind IN ('writer','tribute'))` discriminator** (polymorphic soft-ref, matching the schema's existing polymorphic target grammar `target_protocol`/`resolved_account_id`); `author_return_payout_id` is renamed/folded into this pair (it was the writer-kind case). The carve and reconcile read the kind to know which payouts table to join. *(Considered and rejected for the build session: two strict-FK nullable columns `author_return_payout_id` + `parent_return_tribute_payout_id` with an exactly-one CHECK — keeps strict FKs but reintroduces the parallel author-vs-inspirer column the ADR ruled out; the discriminator keeps the mechanism uniform end-to-end.)*
- **Indexes:** `(parent_tribute_id)` on `tributes` (the child-listing + parent-scoped ceiling lookup); keep the released/swept-unclaimed partial indexes on `tribute_accruals` (the per-cycle reselection is unchanged in shape).
- Regenerate `schema.sql` + re-append the `_migrations` seed in one step; run `scripts/check-schema-drift.sh` (all four checks green).

### Authoring / authorisation

- **`POST /tributes` accepts an optional `parentTributeId`.** The ownership check generalises from "owns the article" (root) to "owns the article **or** is the live inspirer (the `resolved_account_id`) of the parent tribute" (child). On a child, set `author_account_id` ← parent's beneficiary, `article_id` ← parent's article (the whole chain rides one piece, D1), `depth` ← parent.depth + 1 (reject > 8).
- **Parent-scoped serialisation:** the existing per-article `pg_advisory_xact_lock` generalises — roots lock on the article id, children lock on the `parent_tribute_id` — so concurrent adds under one parent can't both read under-ceiling and commit.
- **The combined consent path (C1).** Only a `live` tribute may spawn children (you accept your share before you may pass part of it on — preserves the C6 invariant that every held share traces up an unbroken chain of *consented* earners). Add the third consent option **"Accept & pass a share upstream"** to `POST /tributes/:id/consent`: it consents (`proposed → live`, releasing held accruals exactly as today) **and** returns enough context for the client to open a child-offer composer. No author re-consent is sought when a chain extends below a root (C2 — the author is economically indifferent to the chain's shape below a root; the per-parent ceiling guarantees the author's already-consented outflow is only *partitioned*, never enlarged).

### Money — the recursive generalisation of Phase 3

- **Settlement: a recursive gross-accrual walk** (`confirmSettlement`, still gated on `tributesEnabled()`). Phase 3's single `INSERT … SELECT` (one frozen accrual per (root tribute, settled read) = `FLOOR(read_net × bps / 10000)`) becomes a `WITH RECURSIVE` over the tribute tree ⋈ the just-settled reads: each **node** freezes one `tribute_accruals` row holding its **gross** inflow for that read = `FLOOR(read_net × (∏ bps along the node's path) ÷ 10000^depth)`. Bounded by the depth cap; `ON CONFLICT DO NOTHING` keeps webhook-retry idempotency. State is `released` when the node's tribute is already `live`, else `held` — unchanged per-node rule.
  - **Freeze gross, not net-retained** — this is what lets a child added *after* a read settled carve correctly without rewriting frozen rows (identical to why the author's net is computed live at payout). Accrual is never retroactive: a node only carves reads settled after its child existed, so it keeps the full share on pre-child reads and shares only from then on (exactly as a root tribute never claws back already-settled reads).
  - **Rounding / dust:** each node's gross is an independent per-row `FLOOR`, matching the shipped per-row-then-floor rule (`shared/src/lib/per-read-net.ts`). Each carving party absorbs its own children's flooring dust (the author already absorbs root dust; this generalises "platform/author absorbs dust" to "each node absorbs its direct children's dust"). Conservation telescopes: `node_gross = node_retained + Σ children_gross` at every node, so `author + Σ(every node's retained net) == read_net`.
- **Carve becomes level-aware** (`runPayoutCycle` / `reserveWriterPayout` for the author at depth 0; `runTributePayoutCycle` for every inspirer node). The carve subtracts **direct children regardless of state** — the *exact* rule the author already runs, now applied per carving party:
  - **Author (depth 0)** carves **root** accruals only (was: Σ ALL accruals → becomes Σ accruals whose tribute has `parent_tribute_id IS NULL`). Each root child's disposition is the second leg: `released → paid` to the inspirer / `swept → returned` to the author — unchanged.
  - **Each inspirer node** is now paid `node_gross_released − Σ(its direct children's gross, all states)`, with each direct child's disposition as the second leg: `released → paid` to that child via *its own* `tribute_payouts` run; `swept → returned` to **this** node (C5) folded into *this* node's `tribute_payouts` run. So `runTributePayoutCycle` gains exactly the author's carve+swept-return shape, one level down. This is the bulk of the money work.
  - The shared `shared/src/lib/per-read-net.ts` helper (`perReadNetPence` / `readNetSql`) is the base read-net and **does not change** — what changes is the accrual-aggregation layer wrapped around it (root-vs-direct-children selection) across the money paths and the ~12 net-reading display sites.
- **C5 swept-return-to-an-inspirer-parent — the one new plumbing.** `runTributePayoutCycle` must fold a child's `swept` shares (owed back to *this* node) into this node's payout, claiming them by `swept_return_payout_id` (kind `'tribute'`) and advancing `swept → returned` at complete, rolling back on a failed transfer — the mirror of how `runPayoutCycle` already folds swept returns into the author's `writer_payouts` (kind `'writer'`). The depth-0/author case keeps the writer-kind path; deeper nodes use the tribute-kind path. **One generic return path, read off the discriminator.**
- **Author *and now inspirer* display must carve** (don't ship the gap). Phase 3 made the author dashboard subtract `held|released|paid` accruals; now (a) the author subtracts only **root** `held|released|paid` accruals, and (b) an **inspirer node** with children becomes a co-earner whose own dashboard must subtract its *direct children's* `held|released|paid` from its gross, and whose "reserved, pending redirect" indicator (`getWriterEarnings.reservedPence`, compliance condition #4) shows its in-flight onward carve. Route these through the same accrual-aggregation layer so display and money converge per node.
