## P0 — Live in production now (Phase 1, real money on the tab)

- [x] **F1 · Dispute uniqueness — repeated stakes + public count manipulation.** ✅ 2026-06-24
  `gateway/src/routes/upstream-edges.ts:329` (`POST /disputes`) does no "already disputed?" check and `dispute_edges` (migration 125) has no unique constraint. A third party can dispute the same edge N times (£5 self-charged each — self-harm, but each bumps the public `thirdPartyCount`); a **cited author disputes for free, unlimited** (`is_by_cited_author` ⇒ no stake), inflating the disclaimer/dispute counts the GET endpoints render.
  **Fix:** partial unique index on `(disputant_account_id, citation_edge_id)` and `(disputant_account_id, credit_edge_id)` (each `WHERE … IS NOT NULL AND withdrawn_at IS NULL AND deleted_at IS NULL`), + an idempotent insert (`ON CONFLICT DO NOTHING` → 409/return-existing). New migration (129).
  **As built:** migration `129_dispute_uniqueness.sql` adds the two partial unique indexes (`uq_dispute_active_citation` / `uq_dispute_active_credit`). The insert is now `ON CONFLICT DO NOTHING`; `rowCount===0` skips the stake + relay enqueue inside the txn and returns **409** with the disputant's existing live dispute id (`{ error, id, staked }`). Re-disputing after a withdrawal stays allowed (partial index excludes `withdrawn_at`/`deleted_at`). Frontend `UpstreamEdges.tsx` treats the 409 idempotently (reloads to surface the existing dispute + its Withdraw, no retry prompt). `schema.sql` regenerated; all four drift checks green.

- [x] **F2 · Credit privilege ignores ATProto/ActivityPub/RSS cited parties.** ✅ 2026-06-24
  `gateway/src/routes/upstream-edges.ts:375-380` grants the no-stake privilege only to native members and Nostr-keyed identities. Someone credited via a Bluesky DID / AP actor must pay £5 to disclaim a credit **about themselves** — contradicts the ADR ("the cited author … stakes nothing; anyone else holds a stake"). 
  **Fix:** extend the `is_by_cited_author` match to compare the disputant's linked external identities (`network_presences` / `external_identity_links`) against `credit_edges.target_protocol`/`target_external_id`. If that's too broad for now, document the limitation in the ADR rather than leave it silent.
  **As built:** the credit-dispute branch now adds a third privilege arm — if the credit target is an external protocol, it grants `is_by_cited_author` when the disputant holds an **active, valid** `network_presences` row matching `(protocol, external_id)` exactly. Exact-match only (no fuzzy handle/DID normalisation), so it can never over-grant the free dispute; a format mismatch simply falls back to the £5 stake (the pre-existing behaviour), never an exploit. `external_identity_links` was not needed — it links external↔external author records, not account↔external identity, which is what `network_presences` already provides.

## P1 — Fix before the money flag flips (dark today)

- [x] **F3 · Refund/chargeback unwind is required by the ADR but unbuilt.** ✅ BUILT 2026-06-24 (clears flag-gate (c))
  ADR *Edge cases* and Build-Plan Phase 3 both require a reversed read to "unwind the author's credit **and** its tribute accruals." There was no reader-chargeback/refund path anywhere on the platform — `webhook.ts` handled only `payment_intent.*` / `transfer.*` / `account.updated`; a reader Stripe chargeback after settlement left accruals (and, once paid, real Connect transfers) un-reversed. (Built per the audit's sanctioned shape: platform-wide reversal first, tribute-subtree void composed on top.)
  **As built** (decisions confirmed with the owner: reverse on dispute-**lost** + **refund**, **restore the reader's debt**, **include upvotes**):
  - **Migrations 132–133.** `read_state` += `charged_back`; `tribute_accruals` state CHECK += `voided`; `tab_settlements` += `reversed_at`/`reversal_reason` (the idempotency claim); `vote_charges` += `tab_settlement_id` (FK + index + deterministic earliest-completed-settlement backfill; `confirmSettlement` now stamps it going forward); `ledger_reader_balance` widened with `tab_settlement_reversal` (**mandatory** for −SUM==balance) and `ledger_writer_earnings` with the two payout-reversal triggers.
  - **`shared/lib/ledger.ts`** += `tab_settlement_reversal` / `writer_payout_reversal` / `tribute_payout_reversal`.
  - **`settlement.reverseSettlement(chargeId, reason)`** — one txn: lock tab, idempotently claim `reversed_at`, restore the reader's debt (`balance += amount` + `tab_settlement_reversal` −amount), then apply the pure planner's verdict — flip the charge's reads + vote_charges to `charged_back`, void unpaid (`held|released|swept` unclaimed) accruals, and post per-recipient reversal entries for what already left the platform. No `tributesEnabled()` gate — it is data-driven (zero accruals ⇒ the platform-wide case), so it stays correct across a flag toggled on then off.
  - **`payment-service/src/services/chargeback.ts`** — the pure `computeChargebackReversal` core (repo idiom, cf. `computePublicationSplits`). Per charged-back read: writer_paid → reverse `read_net − Σ root accruals`; `paid` accrual → reverse `node_gross − Σ direct-children gross` against the inspirer; `returned` → reverse against its recipient (author via writer, parent via tribute). Telescopes to exactly −`read_net` for a fully-resolved read; money never paid out is **prevented** (voided / charged-back), not reversed. Proven by `tests/chargeback-reversal.test.ts` (8 conservation cases). The `paid`/`returned` accrual **states are left intact** (the transfer really happened; the reversal is a separate ledger fact) so reconcile A4/A9/A10a/A11 stay green untouched.
  - **`webhook.ts`** — `charge.dispute.closed` (only `status='lost'` — no action on `created`, no re-apply path) and `charge.refunded` (full-amount only; partial logged + skipped).
  - **Guards:** `check-ledger-adjacency.sh` settlement floor → 2; `reconcile-ledger.sql` A6 orphan branch + new **A12** (reversed-settlement ↔ `tab_settlement_reversal` pairing). schema.sql regenerated (throwaway-from-committed); all four drift checks + the payment-service vitest suite green.
  - **Residual (now closed by F17, 2026-06-25):** an accrual a payout had *already reserved* but not yet paid at the instant a chargeback landed (the chargeback-during-payout-cycle window) was originally **skipped** — neither voided nor reversed — letting the in-flight payout pay out clawed-back money with no reversing entry. **F17** closes it: a claimed in-flight accrual is now reversed *as if* it reached its terminal state. See the second-audit-pass section below.

- [x] **F7 · Reminder token rotation can strand a claim.** ✅ 2026-06-24
  `gateway/src/lib/tribute-sweep.ts:72-77` rotated `invite_token_hash` in a committed `pool.query`, *then* emailed the new link; if `sendReminderEmail` threw (swallowed at :83), the **old link was already dead and the new one never delivered**, and `reminder_sent_at` stayed NULL so it rotated again next tick (perpetually killing the prior link).
  **As built:** reordered to send-then-persist. The new token is generated, `sendReminderEmail` is awaited FIRST, and only on a confirmed send does a single `UPDATE` persist the new `invite_token_hash` **and** stamp `reminder_sent_at` together (`WHERE … reminder_sent_at IS NULL`). If the send throws, nothing is persisted: the prior token stays live and the row retries next tick with a fresh token, never stranded. The member/already-claimed branch is an explicit `else` that only stamps `reminder_sent_at`. The advisory-lock half of the finding was already satisfied — `runTributeSweep` runs entirely inside `withAdvisoryLock(LOCK_TRIBUTES, …)` (`gateway/src/index.ts:426/457`) using non-blocking `pg_try_advisory_lock`, so an overlapping tick skips cleanly rather than double-rotating.

- [x] **F11 · Claim token left in the URL.** ✅ 2026-06-24
  `web/src/app/tribute/claim/page.tsx` stashed `?token=…` to `sessionStorage` but never `replaceState`d it out — it persisted in browser history, server access logs, and `Referer`. For a token that binds money to an account, that's a leak.
  **As built:** the stash effect now calls `router.replace('/tribute/claim')` immediately after writing the token to `sessionStorage`. The token is already captured in component state + sessionStorage, so the page keeps working after the URL is cleaned; `router` added to the effect deps.

- [x] **F10 · Citation marker can mis-anchor after a body edit.** ✅ 2026-06-24
  `web/src/components/article/UpstreamEdges.tsx` + `web/src/lib/citation-anchor.ts` placed the marker at the stored `char_start` with **no check that the text there still equalled the stored `excerpt`**. Native articles are NIP-23 replaceable/editable; after a republish the offset is stale and the "[N]" marker silently lands on unrelated prose. Phase 1 (citations) is live, so this was a present correctness gap.
  **As built:** `citation-anchor.ts` gains `excerptMatchesAt(root, start, end, excerpt)` — it walks the same text-node basis (`textBetween`, markers excluded) over `[charStart, charEnd)` and compares the **trimmed** live text to the stored excerpt. (Verified the exact contract: QuoteSelector stores `sel.toString().trim()`; the CitationComposer only sends offsets when the excerpt is unchanged from that span — so an anchored citation's excerpt is always the trimmed selection text. Offsets span the full untrimmed selection, hence the trim on the live side.) The marker-injection effect now filters anchored citations through `excerptMatchesAt` (run on the clean, marker-free root before any insertion splits the basis); a mismatch or an unreachable span drops the marker, the foot apparatus still lists the citation. Web `tsc` clean.

- [x] **F4 · `tribute_accruals` lacks the append-only guard its siblings carry.** ✅ 2026-06-24
  `ledger_entries` is DB-guarded against UPDATE/DELETE/TRUNCATE (migrations 119/124); `tribute_accruals` is money-bearing and its `amount_pence` is documented "frozen … never recomputed" but was freely mutable. (The table *must* allow `state` transitions, so this is a PARTIAL guard, not full append-only.)
  **As built:** migration `130_tribute_accruals_append_only.sql` adds `tribute_accruals_protect()` bound by two triggers — a `BEFORE UPDATE OR DELETE … FOR EACH ROW` and a `BEFORE TRUNCATE … FOR EACH STATEMENT` (mirroring the 119/124 ledger pattern). DELETE/TRUNCATE raise outright; an UPDATE raises if `amount_pence`, `tribute_id`, or `read_event_id` changes (`IS DISTINCT FROM`), and otherwise returns NEW so `state` + the payout-claim columns stay writable. Verified in a throwaway DB: state advance ✓, claim-column write ✓ (reaches its FK check), amount/identity change blocked ✓, DELETE/TRUNCATE blocked ✓. `schema.sql` regenerated (throwaway-from-committed, not a dev-DB dump — dev DB has tribute drift); all four drift checks green.

## P2 — Defense-in-depth / hardening (route-enforced today, no live exploit found)

- [x] **F5 · No DB backstop that a child's parent stays `live`.** ✅ 2026-06-24
  ADR C6's compliance posture leans on "every held share traces up an unbroken chain of consented earners." Verified currently unreachable to violate: `live` is terminal (consent/decline both guard `WHERE status='proposed'`; `DELETE` withdraws proposed only) and children require a live parent (C1) — only a manual DB edit could orphan a child.
  **As built:** migration `131` adds clause (3) to `tributes_validate_write()` (the BEFORE INSERT/UPDATE trigger that already owns the ceiling + D1 invariants): a reserving row with a non-NULL `parent_tribute_id` must have a parent that is `status='live' AND deleted_at IS NULL`, else `check_violation`. Matches the route's `loadLiveParent` ("live and yours") exactly; declined/lapsed/deleted children early-return before the check (no constraint imposed), and a child's own proposed→live consent re-checks against the still-live parent. CREATE OR REPLACE preserving the migration-128 ceiling/D1 body.

- [x] **F6 · Missing consistency CHECKs that sibling tables have.** ✅ 2026-06-24
  `credit_edges`/`tributes` carry `*_target_consistency` CHECKs; this adds the analogues (migration `131`):
  - **`citation_edges_source_consistency`** (immediate CHECK) — mirrors the route's source-column population: a `nostr_external` source must carry `source_author_pubkey`, and a non-Nostr external (atproto/activitypub/rss/email) carries none of the Nostr addressing columns (`source_author_pubkey`/`source_naddr`/`nostr_event_id`/`nostr_d_tag` all NULL). Native (NULL protocol) is unconstrained (member→pubkey or unaddressable→uri both pass). Truth-tabled over six representative rows in a throwaway DB — rejects exactly the two malformed shapes, passes all legitimate ones.
  - **`dispute_edges_wider_excerpt_consistency`** (immediate CHECK) — `(wider_excerpt IS NULL) = (wider_excerpt_sha256 IS NULL)` (the optional re-pin + its hash are all-or-nothing).
  - **Third-party stake presence** — `is_by_cited_author = false ⇒ stake_ledger_entry_id IS NOT NULL` **cannot** be a row CHECK: the stake ledger entry references the dispute and the dispute references the entry (circular), so the route inserts the dispute first (stake NULL) then back-fills it in a second UPDATE in the same txn — the invariant only holds at COMMIT. Implemented as the **DEFERRABLE INITIALLY DEFERRED constraint trigger** `trg_dispute_edges_check_stake` (Postgres has no deferrable CHECK), which fires at commit after the back-fill. Withdrawal retains `stake_ledger_entry_id`, so it holds for withdrawn rows too.
  - **Note:** `citation_edges` is live in production (Phase 1); the new CHECK validates existing rows on apply. The route has populated the source columns consistently, so this is expected to pass, but a pre-existing malformed row would surface here (intended).

- [x] **F8 · `consent`/`decline`/`claim` lack the `23514` catch that `create` has.** ✅ 2026-06-24
  `gateway/src/routes/tributes.ts` ran the consent/decline/claim UPDATEs (tribute writes firing the ceiling/D1/now-parent-live trigger) with no try/catch → a `check_violation` would surface as a 500. Verified **not reachable today** (insert-time enforcement keeps Σ proposed+live ≤ 9000; the reverse trigger blocks publication-while-tributed; decline early-returns in the trigger) — latent robustness only, made marginally more relevant by F5's new parent-live arm.
  **As built:** all three handlers' `withTransaction` calls are wrapped in `try/catch`; a `23514` returns a 400 with a handler-appropriate message (consent/decline/claim "can no longer be …"), every other error re-throws. Mirrors the `create` handler's existing catch. Gateway `tsc` clean.

- [ ] **F9 · Self-tribute is not blocked** (`gateway/src/routes/tributes.ts`, `POST /tributes`, no `t.accountId !== writerId` check). **By design** — the ADR documents "Self-crediting is not uplift … Noted as a limit, not plugged," and conservation still holds. Listed only so it isn't re-discovered as a "bug." Add a guard only if the circle polluting the accrual/ledger model becomes a concern.

- [x] **F13 · `UpstreamEdges.tsx` robustness.** ✅ 2026-06-24. All three sub-issues fixed in `renderTributeForest` / `acceptAndPass`: (1) **visited-set guard** — `walk` carries a `seen: Set<string>` and renders each id at most once, so a malformed/cyclic payload can never recurse forever; (2) **orphans re-rooted** — a child whose `parentTributeId` is absent from the returned payload is grouped under `null` (an `ids` set gates the parent key), so its share still renders; plus a trailing sweep force-renders any node a pure cycle kept off every root path, so no consented share is ever hidden; (3) **double-accept closed** — `acceptAndPass` treats `consent()` as the commit point and wraps the subsequent `load()` in its own `try/catch`, so a post-consent reload failure no longer propagates to the button's `.catch` (which would surface "try again" and re-consent an already-live tribute). A stale row is tolerated; a double-accept is not.

## Second audit pass (2026-06-25) — payment-service failure/confirmation edges

A four-agent re-audit of the three-day tribute body of work (ledger/migrations, payment-service, gateway routes, frontend). Ledger core, gateway routes, and frontend came back clean (two LOW nits only). The material risks all sat on the **failure/confirmation edges** the happy-path tests don't exercise — paths added piecemeal as the new money flows landed but never extended past the writer case. All four fixed below.

- [x] **F15 · Tribute & publication transfers had no `transfer.paid`/`transfer.failed` handling.** ✅ 2026-06-25 (HIGH)
  `webhook.ts` routed `transfer.paid`→`confirmPayout` and `transfer.failed`→`handleFailedPayout`, but **both only `UPDATE writer_payouts`** (`payout.ts:707/734`). Tribute (`payout.ts:1417`) and publication-split (`:1092`) transfers are real `transfers.create` calls that emit the same webhooks, so they matched no row, hit the log-and-return no-op, and: a landed tribute transfer never advanced past `initiated` (no `completed` state existed on `tribute_payouts` at all), and a **failed** tribute/pub transfer was never rolled back — the inspirer's accruals stayed `paid` and the `+tribute_payout` ledger entry stood for money that never landed, with no re-pay.
  **As built:** migration `134_payout_confirm_lifecycle.sql` adds `completed` + `completed_at` to `tribute_payouts` (pub already uses the `payout_status` enum). `webhook.ts` now routes both transfer events by `transfer.metadata` (`tribute_payout_id` / `publication_payout_id`). New `payout.ts::confirmTributePayout` + `handleFailedTributePayout` (rolls accruals `paid→released` / swept-returns `returned→swept`, unclaims, so the next cycle re-pays under a **fresh** row — new id ⇒ new idempotency key ⇒ a genuinely new transfer, mirroring the writer path) and `confirmPublicationSplit` + `handleFailedPublicationSplit` (split → `completed`/`failed`; parent → `completed` only when *every* split has). `reconcile-ledger.sql` **A10a/A10b** now exclude `failed` tribute_payouts (the failed payout's append-only `+` entry stays, same reconciliation-only posture as the writer path). **Deferred (feature-debt.md):** publication-split *auto re-pay* — a failed split is now visible but not auto-retried (the stable idempotency key would dedupe to the failed transfer; a correct retry must mint a fresh split row).

- [x] **F16 · No backstop for a dropped `payment_intent.succeeded`.** ✅ 2026-06-25 (HIGH)
  `completeSettlement` flips a settlement to `completed` the instant the PaymentIntent is created (card charged), but the tab debit, reader ledger credit, and read/vote/tribute advancement all run in `confirmSettlement` — driven **only** by the `payment_intent.succeeded` webhook. Stripe delivers at-least-once, not always-once: one dropped event left the reader **charged** with no tab/ledger movement and reads stuck `accrued` forever, with no error. KYC got a reconcile sweep (F-prog 2026-06-24) but settlement had none.
  **As built:** `settlement.ts::reconcileSettlements` + `workers/settlement-reconcile.ts` (3×/day at 00:15/08:15/16:15 UTC, wired in `index.ts`) — the settlement twin of `reconcileConnectKyc`. Candidate = a `completed` settlement with `stripe_charge_id IS NULL` older than a 1h grace; re-reads the PaymentIntent from Stripe and, if `succeeded`, calls the (idempotent) `confirmSettlement`. A flip the webhook also lands is a safe no-op (charge-claim guard).

- [x] **F17 · Chargeback racing an in-flight payout could create money.** ✅ 2026-06-25 (MEDIUM — supersedes the F3 residual)
  `chargeback.ts` **skipped** a `claimed` (a payout had reserved it) `held|released|swept` accrual. But a claimed accrual is in-flight: the reserved payout WILL pay/return it. Skipping it let the payout pay out clawed-back money with **no reversing entry** → money created (the inspirer keeps cash the reader recovered, ledger never records the platform absorbing it).
  **As built:** the planner now reverses a claimed in-flight accrual *as if* it reached its terminal state — `released` claimed via `tribute_payout_id` ⇒ treat as `paid`; swept claimed via `swept_return_payout_id` ⇒ treat as `returned` (the `sweptReturnKind` discriminator picks the branch). Conservation still nets to `−read_net`. UNclaimed accruals are still voided. Four new `chargeback-reversal.test.ts` cases lock it (claimed-released, claimed-swept kind writer/tribute, and the UNclaimed-still-voided control). **New, far narrower residual:** if the in-flight payout's transfer ultimately *fails* or stays permanently pending (inspirer lost KYC), the reversal slightly over-reports the platform's loss — a reconciliation-only artefact, never user-facing phantom money. (And F15's tribute re-pay actually self-heals the failed-transfer case: the re-pay posts the matching `+`.)

- [x] **F18 · `confirmSettlement` threw on an unknown PaymentIntent.** ✅ 2026-06-25 (MEDIUM)
  `settlement.ts:382` threw `No settlement found for PaymentIntent` on no match → the webhook's 500 path left `stripe_webhook_events.processed_at` NULL, so Stripe redelivered the same event for days, each retry re-throwing (a poison event). Today every PI is a settlement so it was latent, but any out-of-band PI would poison the queue.
  **As built:** `rowCount===0` now logs a warning and returns, matching the no-match handling in `reverseSettlement` and `handleFailedPayment`.

**Verified green (2026-06-25):** payment-service `tsc` exit 0 + vitest **65/65** (was 62; +3 chargeback cases, net of the rewritten F3-residual test) · shared vitest 83/83 · `check-ledger-adjacency.sh` ✓ (no new money-write site; the new handlers flip status / roll back state, no un-mirrored balance write) · `check-schema-drift.sh` all 4 (134 migrations seeded, 314 objects, migrate no-op, round-trip) · root promise-safety lint 0 errors. `reconcile-ledger.sql` not re-run (needs a populated fully-migrated DB; the live dev DB has tribute drift — its A10a/A10b view-structure side is covered by drift Check 1/2).
